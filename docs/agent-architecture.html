Core Thesis

DeepFlow uses a separate runtime per task. Every task in a workflow gets its own execution environment — its own tools, its own permissions, its own context window. This is not how most AI orchestration platforms work, and the choice has profound implications for security, scalability, and architectural complexity.

Most AI agent systems — from personal assistants like OpenClaw to framework-level tools like LangChain — operate in a single runtime. The agent loads once, gets access to everything, and runs until it's done. It's simple, fast, and works beautifully for single-user, trusted environments.

DeepFlow takes a different path. When a workflow decomposes into tasks, each task spawns its own isolated execution environment. A task reviewing financial compliance loads legal and audit tools. A task writing marketing copy loads content and brand tools. Neither sees the other's capabilities. Neither can invoke the other's permissions.

This document examines why that choice matters, when it's the right call, and when it's over-engineering.

1. Single Runtime vs. Per-Task Runtime

The single-runtime model

In a single-runtime system, the AI agent operates like a user sitting at a workstation. It has one session, one set of tools, one permission boundary. OpenClaw is a clean example: one host machine, one agent, access to everything the host provides — files, APIs, shell commands, messaging surfaces.

The mental model is straightforward: the agent sees what you see. If you can read a file, the agent can read it. If you can send an email, so can the agent. Permission boundaries are coarse — they're your operating system's permissions, not the agent's.

This works remarkably well for specific scenarios:

  • Personal assistants. A solo user trusts the agent with their environment. There's no multi-tenancy concern.
  • Developer tooling. Coding agents that need full filesystem access, git operations, and shell execution.
  • Single-operator workflows. One person, one machine, one context.

The strengths are real: low latency (no runtime spin-up), full context (the agent accumulates knowledge across tasks), simple mental model (one agent, one session), and minimal infrastructure overhead.

The per-task runtime model

DeepFlow's model is fundamentally different. When a workflow decomposes a goal into tasks, each task gets its own execution environment. Think of it as Docker containers for AI execution — shared orchestration kernel, isolated userspace per task.

Workflow: "Prepare quarterly board report" ┌─────────────────────────────────────────────────┐ │ Orchestrator │ │ Manages DAG, routes context, enforces policy │ ├─────────────┬───────────────┬───────────────────┤ │ │ │ │ │ Task 1 │ Task 2 │ Task 3 │ │ Financial │ Marketing │ Compliance │ │ Analysis │ Summary │ Review │ │ │ │ │ │ Tools: │ Tools: │ Tools: │ │ · SQL │ · CMS API │ · Audit log │ │ · Excel │ · Analytics │ · Policy docs │ │ · Charts │ · Brand kit │ · Reg database │ │ │ │ │ │ Perms: │ Perms: │ Perms: │ │ · Finance │ · Marketing │ · Compliance │ │ data (R) │ data (R) │ data (R/W) │ │ · No email │ · No finance │ · Escalation │ │ · No deploy│ · No deploy │ · No deploy │ └─────────────┴───────────────┴───────────────────┘

Task 1 can query the financial database but cannot access the CMS. Task 3 can read compliance policies but cannot touch marketing analytics. Each runtime is configured for exactly what the task needs — nothing more.

The Docker analogy

This maps cleanly to containerisation. In a single-runtime system, the agent runs as a process with the host's full permissions — like running an application directly on bare metal. In DeepFlow's model, each task runs in its own container: shared orchestration kernel, isolated tool access, defined resource limits, and clean teardown.

The analogy extends further:

  • Namespaces → each task has its own tool namespace; no cross-contamination
  • Resource limits → each task has its own context window budget and token ceiling
  • Networking → tasks communicate through the orchestrator, not directly (like container networking through a bridge)
  • Image layers → skill definitions serve as the "image" from which task runtimes are instantiated

When does each model make sense?

FactorSingle RuntimePer-Task Runtime

Data sensitivityUniform sensitivityVariable sensitivity across tasks
Regulatory needsLow / self-regulatedAuditable, per-task traceability
Team sizeSolo or small, trusted teamCross-functional, role-based

Neither model is universally superior. A solo consultant running a personal AI assistant should use a single runtime — isolation adds overhead with no security benefit. A 50-person team running regulated workflows across departments needs per-task isolation — shared access is a compliance failure waiting to happen. DeepFlow's bet is that its target market (structured team workflows) justifies the per-task approach.

2. Tools vs. Skills — A Crucial Distinction

Tools are atomic. Skills are composed.

This distinction sounds semantic but it's architecturally fundamental. Getting it wrong leads to either over-permissioned agents (every task gets every tool) or under-capable ones (tasks can't accomplish their goals without manual intervention).

A tool is a single capability with a defined interface:

  • read_file(path) → returns file contents
  • query_database(sql) → returns rows
  • send_email(to, subject, body) → sends an email
  • call_api(endpoint, method, payload) → returns response

Tools are stateless. They don't know why they're being called. They don't carry context between invocations. They're functions.

A skill is a composed behaviour that orchestrates multiple tools with domain knowledge:

Skill: "Review a Pull Request"

This isn't one tool call. It's a sequence with judgment:

1. Read the diff (tool: read_file)
2. Understand the codebase context (tool: search_files, read_file)
3. Check CI status (tool: check_ci)
4. Evaluate against coding standards (domain knowledge, not a tool)
5. Write review comments (tool: create_review)
6. Approve or request changes (tool: submit_review)

The skill defines the strategy — what to check, in what order, what standards to apply. The tools are just the hands.

Why this matters for per-task runtimes

In DeepFlow's architecture, the skill-tool distinction enables principle of least privilege at the capability level. When a task is assigned, its skill definition determines which tools are loaded into that runtime:

  • A "compliance review" skill loads: audit log reader, policy document search, regulatory database query, report generator. It does NOT load: deployment tools, email sending, file deletion.
  • A "code deployment" skill loads: git operations, CI/CD triggers, environment configuration, health checks. It does NOT load: financial database access, HR systems, customer data queries.
  • A "customer outreach" skill loads: CRM query, email composer, template engine, analytics. It does NOT load: source code access, infrastructure controls, internal wikis.

This is the security model working as intended. The skill is the unit of capability scoping. The tools are the implementation details.

Skill composition in practice

Skills can be nested. A high-level skill like "prepare quarterly report" might decompose into sub-skills, each with their own tool requirements:

Skill: "Prepare Quarterly Report" ├── Sub-skill: Financial Analysis │ ├── Tool: query_database (finance schema only) │ ├── Tool: generate_charts │ └── Tool: format_spreadsheet │ ├── Sub-skill: Executive Summary │ ├── Tool: read_document (previous reports) │ ├── Tool: search_notes (meeting minutes) │ └── Tool: generate_text │ └── Sub-skill: Compliance Check ├── Tool: audit_log_query ├── Tool: policy_lookup └── Tool: flag_exceptions

Each sub-skill runs in its own task runtime with only the tools it needs. The financial analysis task cannot read meeting minutes. The compliance check cannot generate charts. The orchestrator manages the flow of outputs between them.

Skills carry context and strategy

A critical difference: tools are interchangeable (any SQL client will do), but skills encode institutional knowledge. A "compliance review" skill at a fintech startup is fundamentally different from one at a pharmaceutical company — different regulations, different risk thresholds, different escalation paths.

This makes skills the primary vector for customisation in DeepFlow. Users don't configure tools (those are infrastructure); they configure how work gets done through skill definitions. Over time, an organisation's skill library becomes a codified version of "how we do things here" — a genuine competitive asset.

3. Runtime Architecture Deep Dive

Task runtime lifecycle

Every task in a DeepFlow workflow follows the same lifecycle:

Task Runtime Lifecycle SPAWN Orchestrator creates a new runtime instance │ Triggered by: DAG dependency resolution, manual assignment ▼ CONFIGURE Load tools based on skill definition │ Set permissions from user role + task scope │ Inject context from upstream task outputs │ Set resource limits (tokens, time, cost ceiling) ▼ EXECUTE Agent runs within configured boundaries │ Tool invocations logged individually │ Progress reported to orchestrator │ Can request additional context (orchestrator-mediated) ▼ REPORT Output structured and stored │ Downstream tasks notified of availability │ Quality/completeness signals generated ▼ TEARDOWN Runtime destroyed Logs preserved for audit Tools unloaded, permissions revoked Resources released

The key insight: teardown is not optional. Every runtime is ephemeral. There's no persistent agent accumulating permissions over time. Each task starts clean and ends clean. This is a security property, not just an implementation detail.

Shared state vs. isolated state

If every task is isolated, how do tasks in the same workflow communicate? This is the central tension of per-task architecture.

DeepFlow uses an orchestrator-mediated message bus. Tasks don't talk to each other directly. Instead:

  1. Task outputs are structured. When Task A completes, its output is stored as a typed artifact (document, data table, analysis, decision).
  2. The orchestrator routes outputs to dependents. Based on the DAG, Task B receives Task A's output as part of its context injection during the CONFIGURE phase.
  3. The orchestrator controls visibility. Task B might receive Task A's summary but not its raw data. The orchestrator can filter, summarise, or transform outputs before passing them downstream.

This is more restrictive than shared memory but more auditable. Every piece of information that crosses a task boundary is logged, traceable, and filterable.

The trade-off is real. In a single-runtime system, the agent naturally accumulates context across tasks — it remembers what it did two steps ago. In DeepFlow's model, that context must be explicitly passed. If the orchestrator doesn't inject a relevant upstream output, the downstream task operates blind. The system's intelligence is only as good as its context routing.

The context window problem

Every LLM-based agent operates within a finite context window. In a single-runtime system, one long-running agent gradually fills its window with conversation history, tool outputs, and accumulated knowledge. Eventually, important early context falls off the edge.

Per-task runtimes flip this problem. Each task starts with a fresh context window, populated only with what's relevant:

  • Skill instructions — how to approach this type of work
  • Upstream outputs — results from dependent tasks (filtered by the orchestrator)
  • Tool definitions — what capabilities are available
  • Task-specific context — relevant documents, data, constraints

This is genuinely advantageous for complex workflows. A task reviewing compliance doesn't waste context tokens on the marketing task's creative brief. Each task gets a curated, relevant context window instead of an accumulated, noisy one.

But it creates a new problem: the orchestrator must be smart about context curation. Too little context → the task lacks necessary information. Too much → you've recreated the single-runtime bloat problem, just with extra steps. Getting this right is possibly the hardest engineering challenge in DeepFlow's architecture.

Resource management

Per-task runtimes have non-trivial cost implications:

Cost FactorSingle RuntimePer-Task Runtime
LLM API callsOne session, accumulated contextMultiple sessions, duplicated system prompts
Tool loadingLoad once, use for everythingLoad per task, subset each time
Context overheadGrowing but sharedFresh but repeated (skill defs, tool schemas)
OrchestrationMinimal (agent self-orchestrates)Significant (DAG management, context routing)
ParallelismSequential (one agent)Parallel (independent tasks run simultaneously)

The parallelism benefit partially offsets the overhead cost. A workflow with 10 independent tasks completes in the time of the slowest task, not the sum of all tasks. For workflows with natural parallelism (and DAGs often have significant parallelism), per-task runtimes can be faster despite per-task overhead.

Cold start vs. warm pool

Spinning up a fresh runtime for every task introduces latency. The mitigation strategies are well-understood from serverless computing:

  • Warm pool: Keep pre-configured runtimes ready for common skill types. A "code review" runtime is always available; it just needs context injection.
  • Predictive pre-warming: When Task A is 80% complete, pre-warm runtimes for its downstream dependents.
  • Runtime reuse: For tasks with identical skill + tool requirements, reuse a runtime with a context flush (clear prior task's state, inject new context).
  • Tiered latency: Not all tasks are latency-sensitive. Background analysis tasks can tolerate cold starts; interactive chat-based tasks cannot.

The target should be sub-second runtime configuration for warm pools and under 3 seconds for cold starts. Anything longer and users perceive the system as sluggish compared to single-runtime alternatives.

4. Permission Model at the Runtime Level

From UI roles to runtime capabilities

DeepFlow's UI defines four permission levels: Owner, Member, Read-only, and Agent. These are familiar — most SaaS platforms have something similar. What's less common is how these translate into runtime-level tool permissions.

Permission Cascade: UI → Runtime Owner (Sarah) └─ Can spawn tasks with ANY tool in the workspace └─ Can grant/revoke tool access for other roles └─ Runtime: full tool catalogue available Member (Ravi) └─ Can spawn tasks with tools assigned to their role └─ Cannot access Owner-restricted tools (billing, user mgmt) └─ Runtime: scoped tool catalogue Read-only (David the auditor) └─ Can view task outputs and audit logs └─ Cannot spawn tasks or invoke tools └─ Runtime: view-only, no tool execution Agent (AI executing a task) └─ Inherits EXACTLY the permissions of its invoking user └─ Cannot escalate beyond invoker's access level └─ Runtime: subset of invoker's tool catalogue

The critical rule: an AI agent's runtime never exceeds its invoking user's permissions. If Ravi (Member) triggers a workflow, every AI task in that workflow is bounded by Ravi's access level. The agent cannot spawn a task that accesses Owner-restricted tools, even if the skill definition references them.

This is where per-task isolation pays clear dividends. In a single-runtime system, enforcing per-action permission checks requires the agent to self-police (or a middleware to intercept every tool call). In DeepFlow's model, the permissions are baked into the runtime at spawn time. The tools simply aren't available — there's nothing to intercept.

Agent permission inheritance

When an AI agent executes a task, its effective permissions are the intersection of three sets:

  1. Invoking user's role permissions — what the user who triggered the workflow can access
  2. Skill-defined tool requirements — what tools the skill says it needs
  3. Task-specific scope — any additional restrictions set at the workflow level (e.g., "this task can only read from the Q1 dataset")

The result: the smallest reasonable set of capabilities. If a Member invokes a "financial analysis" skill that requests SQL, chart generation, and email tools — but the Member doesn't have email permissions — the runtime loads SQL and chart generation only. The task may produce a less complete output (no emailed report), but it won't violate the permission boundary.

Audit trail

Every tool invocation in a per-task runtime generates an audit record:

  • Who — which user triggered the workflow
  • What — which tool was called, with what arguments
  • When — timestamp
  • Where — which task, which workflow, which runtime instance
  • Why — which skill definition requested this tool
  • Result — what the tool returned (or that it failed)

This is table-stakes for regulated industries, but it's also valuable for debugging. When a workflow produces unexpected output, the audit trail shows exactly which tool call in which task generated the problematic data. In a single-runtime system, you get a log of tool calls, but tracing them to specific logical tasks requires inference. In DeepFlow's model, the tracing is structural.

Key takeaways from the comparison

DeepFlow is the only platform with genuine runtime isolation. CrewAI and AutoGen have conceptual agent separation (different roles, different tools), but they share a process. A misconfigured tool in one agent can leak data to another. DeepFlow's isolation is structural, not conventional.

LangChain/LangGraph is the closest architectural competitor — not in isolation (it has none) but in graph-based execution. LangGraph's "state graph" model is conceptually similar to DeepFlow's DAG, but without the permission layer, audit trail, or runtime isolation. It's a developer framework; DeepFlow is a platform.

OpenClaw occupies a genuinely different niche. It's not competing with DeepFlow — it's solving a different problem. A personal assistant that knows your entire environment is the opposite of an isolated task runtime. Both are valid; they serve different needs.

DeepFlow's architectural differentiation is real — per-task isolation with permission inheritance and structured audit trails doesn't exist in any current competitor. The question is whether the market values this differentiation enough to absorb the complexity cost.

6. Recommendations

What DeepFlow should prioritise

Priority 01

Get context routing right before anything else

The orchestrator's ability to curate and route the right context to each task runtime is the make-or-break capability. Too little context → useless outputs. Too much → expensive and noisy. Invest in smart summarisation, relevance scoring, and user-configurable context policies. This is harder than the isolation model itself.

Priority 02

Build warm pools for common skill types

Cold-start latency will kill adoption for interactive workflows. Pre-warm runtimes for the top 10 skill types. Use predictive pre-warming based on DAG structure. Target sub-second task start for warm runtimes. Users will forgive many things; they won't forgive feeling slow.

Priority 03

Make the permission model invisible until it matters

Most users won't care about per-task tool isolation — until they do. Design the permission system to be zero-configuration for solo users (everything works, no friction) and progressively configurable for teams. Don't make Ravi learn about permission inheritance to use the product. Let Sarah configure it when her team needs it.

Priority 04

Ship a "shared runtime" mode for simple workflows

Not every workflow needs per-task isolation. A solo user running a personal task list gains nothing from spawning isolated runtimes. Offer a lightweight "shared runtime" mode for low-complexity workflows — one agent, accumulated context, all tools available. Switch to per-task isolation when the workflow involves multiple users, sensitive data, or governance requirements.

Where per-task isolation adds genuine value

  • Multi-department workflows. Finance + Legal + Marketing tasks in one workflow — isolation prevents cross-contamination of sensitive data.
  • Regulated industries. Per-task audit trails satisfy compliance requirements that no other AI platform currently meets.
  • Mixed human/AI execution. When AI tasks and human tasks interleave, per-task permissions ensure the AI never exceeds the human's access level.
  • Parallel execution. Independent tasks running simultaneously is only possible with isolated runtimes. A single-runtime agent must execute sequentially.

Where it adds unnecessary complexity

  • Solo user, single-domain workflows. One person managing their own tasks doesn't need isolation. The overhead is pure cost.
  • Rapid iteration / exploratory work. When a user is brainstorming and quickly spawning tasks, runtime spin-up latency disrupts flow.
  • Small teams with full trust. A 5-person startup where everyone has access to everything gains governance overhead without security benefit.

The "good enough" permission model

For most teams under 20 people, three permission levels suffice:

1. Admin — can configure tools, skills, and permissions
2. User — can trigger workflows and execute tasks with default tool sets
3. Viewer — can see outputs and audit trails, cannot execute

Don't ship a complex RBAC system on day one. Ship these three levels, make them work flawlessly, and add granularity when enterprise customers demand it with purchase orders attached.

8. Self-Critique

An architecture document that doesn't question its own assumptions is marketing, not analysis.

Is per-task runtime over-engineered for most use cases?

Probably, yes. The honest assessment: 80% of AI agent workflows today are single-user, single-domain, moderate-complexity tasks. A developer asking an AI to review code. A marketer generating campaign copy. A manager summarising meeting notes. None of these need per-task isolation.

DeepFlow's architecture is designed for the 20% — multi-user, cross-domain, regulated workflows where isolation is a genuine requirement. That's a real market (enterprise, finance, healthcare, legal), but it's not the mass market. If DeepFlow targets the 80% with an architecture designed for the 20%, it will feel heavy. The "shared runtime" mode recommended above isn't optional — it's existential for adoption.

The performance cost of isolation

It's non-trivial and shouldn't be hand-waved. Every per-task runtime incurs:

  • System prompt duplication (skill instructions, tool schemas) — ~500-2000 tokens per task
  • Context injection overhead — serialising and deserialising upstream outputs
  • Cold start latency — even with warm pools, configuration takes time
  • Orchestration cost — the DAG engine, context router, and permission validator are all additional compute

For a workflow with 20 tasks, the overhead could be 10,000-40,000 additional tokens compared to a single-runtime approach. At current API pricing, that's small in absolute terms but compounds at scale. More importantly, the latency adds up — 20 tasks with 1-second cold starts is 20 seconds of pure overhead before any work begins (mitigated by parallelism, but still present on the critical path).

When shared state is actually what you want

Isolation is the wrong default for collaborative, real-time work. Consider:

  • Collaborative document editing. Multiple agents contributing to the same document need shared state, not isolated runtimes passing summaries through an orchestrator.
  • Real-time dashboards. A monitoring workflow where tasks need instant access to each other's latest data — orchestrator-mediated message passing adds unacceptable latency.
  • Conversational agents. A customer support workflow where context from one exchange must immediately inform the next — isolation destroys conversational coherence.
  • Exploratory data analysis. An analyst asking iterative questions about a dataset — each question depends on seeing what the last answer revealed. Per-task isolation means losing the analytical thread.

DeepFlow must recognise that isolation is a tool, not a philosophy. Some workflows benefit from shared state. The architecture should support both modes gracefully.

The complexity tax: is this worth it for teams under 20 people?

Almost certainly not as the default experience. A team of 8 people will never configure per-task tool permissions. They'll never define skill-level capability scopes. They'll never review per-runtime audit trails. They'll use the product like a project management tool with AI features — and if the per-task architecture makes that experience slower or more confusing than Asana with an AI plugin, they'll choose Asana.

The complexity tax is not just technical — it's cognitive. Every concept a user must understand (runtimes, tools vs. skills, permission inheritance, context routing) is a barrier to adoption. The architecture should be invisible to small teams and only surface for organisations that need it.

The test: Can a solo user run DeepFlow without ever encountering the word "runtime"? If yes, the architecture is well-hidden. If no, it's leaking implementation into UX.

Bottom line: Per-task runtime isolation is a genuinely differentiated architecture with real value for regulated, multi-user workflows. But it's a liability if it's the only mode. DeepFlow needs to be simple by default and powerful by configuration — not powerful by default and simple never. The architecture should serve the product, not the other way around.

DeepFlow's per-task runtime model is architecturally novel in the AI orchestration space. No current competitor offers genuine runtime isolation with permission inheritance and structured audit trails. The question isn't whether the architecture is sound — it is. The question is whether the product can hide the complexity from users who don't need it while exposing it to users who do.

The best infrastructure is invisible. Make the runtimes invisible.
