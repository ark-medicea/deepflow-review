01

Brand Overview

What DeepFlow is, who it serves, and the single idea every design decision flows from.

Describe what you want. DeepFlow figures out how to get it done.

DeepFlow is an AI orchestration platform that transforms natural language into executable task graphs — dependency-aware, auditable, and governed. It bridges the gap between "I need this done" and the structured, accountable process that gets it done, coordinating humans and AI agents side by side.

🎯
Orchestration, Not Automation

DeepFlow doesn't replace people — it coordinates them. Humans and AI work the same graph, with clear handoffs, dependencies, and accountability.

🔍
Governed Transparency

Every action is timestamped, attributed, and auditable. Built for industries where "who did what and when" isn't optional — it's regulation.

⚡
Power Without Complexity

Enterprise capability, approachable interface. The user describes intent in plain language; the platform handles decomposition, sequencing, and execution.

Gradient on Dark — Primary
White on Gradient — Hero / Marketing
Brand Name Usage

The wordmark logo renders "deepflow" in lowercase as a single word — this is correct for the logo image. When used as text (in UI labels, documentation, marketing copy, or conversation), always write DeepFlow with capital D and F for legibility. Never "Deepflow", "deepflow", or "Deep Flow" in running text.

02

Design Principles

Six rules that every design decision must pass through. When in doubt, refer here.

01
Calm by Default, Vivid by Exception

The interface is monochrome. Colour is reserved for information that demands attention: status changes, actionable items, brand moments. If everything is colourful, nothing is.

In practice → A graph of 50 nodes is rendered in neutral grays. The 3 nodes that need attention glow with status colour. Your eye goes exactly where it should.
02
Show the Structure, Hide the Complexity

DeepFlow manages intricate dependency graphs. The UI's job is to make that structure legible — not to expose every node, edge, and metadata field at once. Progressive disclosure: show what matters now, reveal depth on demand.

In practice → A collapsed parent task shows its name, status, and child count. Expanding it reveals children in a shaded containment area. You drill down when you choose to.
03
Status Is Sacred

In an orchestration platform, status is the most important piece of information. It must be visible, consistent, and unambiguous in every view. One visual language for status, everywhere.

In practice → Green means complete. Red means failed. Amber means needs attention. Always. In every view, at every zoom level. No exceptions.
04
Readable Before Beautiful

Users managing 50+ tasks across workstreams cannot afford to squint. Task names are never truncated to invisibility. Labels are always present alongside icons. Contrast meets WCAG AA. Legibility is non-negotiable.

In practice → If a task name doesn't fit, the container grows — not the text shrinks. Icon-only buttons always have a tooltip, and primary actions always have visible text labels.
05
Client-Presentable by Default

Solo founders screenshare DeepFlow with their clients. Operations managers present dashboards to executives. The UI must look trustworthy, polished, and professional at all times — not like a developer tool. If a client sees it, they should think "this person has their act together."

In practice → Clean typography, generous whitespace, restrained colour. No exposed technical jargon, no cluttered toolbars. The interface looks like it belongs in a boardroom, not a hackathon.
06
Human and AI, Visually Distinct

DeepFlow uniquely blends human and AI task execution. At a glance, users must know whether a task is assigned to a person or an agent. This distinction is expressed through shape, never colour.

In practice → Circular avatars = human assignees. Hexagonal avatars = AI agents. This shape language persists across graph nodes, list rows, kanban cards, and detail panels.
⸻

The Line

DeepFlow's visual signature. One subtle accent line signals interactivity, state, or focus. Full borders mean static containers. A single line means something is alive.

A single line is the smallest unit of visual information. One line — placed deliberately — tells you this element is interactive, this element has state, or this element has your attention.
The Rule

Full border = static container. A card, a panel, a section boundary. It holds content but doesn't communicate state.

Single line = interactive or stateful element. An editable field, an active navigation item, a status indicator, a notification severity. The line is the signal.

Line Position Encodes Meaning

Bottom line: editable text (inputs), current location (active tab).
Left line: editable block (textarea), element status (cards, nodes, toasts), navigation state (sidebar), hover focus (table rows).
Top line: dialog intent and severity (destructive, action, approval).

Position isn't arbitrary — it follows spatial convention. Bottom = "what I'm typing in." Left = "what this thing is." Top = "what this dialog wants."

Line Colour Encodes State

Green = complete, success.
Yellow/Amber = in progress, active, warning, needs approval.
Blue (brand) = interactive, focused, selected, editable.
Red = failed, error, destructive.
Grey = pending, inactive, disabled.

The line colour always maps to the existing status/semantic palette — no new colours are introduced. The system piggybacks on the tokens already defined.

What Doesn't Get a Line

Not everything needs the treatment. Static information cards, the graph canvas background, sidebar icons, breadcrumb separators, avatar badges — these are structural or decorative. Adding a line would create noise, not signal.

The test: Does this element have interactive state or variable status? If yes, it gets a line. If it's purely structural, leave it alone.

Consistency Is the Point

The power of this system isn't any single line — it's that every line follows the same logic. A user who learns that "left accent = status" on a card will recognise the same pattern on a graph node, a toast, and a table row. One visual grammar, learned once, applied everywhere.

Voice Principles

Clarity Over Cleverness

Users are managing regulated processes. Use plain language. Say "3 tasks are blocked" not "Some tasks need a nudge." Save personality for empty states — never in error messages.

Action-Oriented

Tell the user what to do, not just what happened. "Task failed — retry or reassign" beats "Task failed." Every message implies a next step.

Respect the User's Time

No filler. No "Please note that…" or "It appears that…" Get to the point. Sarah has 50 tasks to manage — every word must earn its place.

Monochrome + pops of colour. The canvas is calm; colour earns its place by meaning something.

The Philosophy

The interface is built on a monochrome neutral scale — dark backgrounds, light text, subtle borders. Colour appears only for three purposes: brand identity (cyan→blue gradient), semantic status (success, warning, error), and parent-child grouping (subtle containment shading). If a colour doesn't serve one of these three roles, it doesn't belong.

The DeepFlow signature. A vibrant sweep from electric cyan through sky blue to deep cobalt. Used for brand moments, hero areas, primary CTAs, and accent highlights.

05b

Gradient vs. Solid Colour

The brand gradient is distinctive but expensive — overuse dilutes its impact. These rules define when to use the gradient and when solid brand colour is more appropriate.

Use the Gradient For

Brand-level moments that should feel premium and distinctive. The gradient says "this is DeepFlow" — it's the visual signature.

  • Logo and iconmark (always gradient or white)
  • Hero sections and landing page headers
  • Primary CTA button (one per view maximum)
  • Onboarding and empty-state illustrations
  • Gradient text for display headings (sparingly)
  • Loading/progress indicators (the animated edge flow on graph)
Use Solid Brand Colour For

Everyday UI elements where brand presence is needed but the gradient would be too heavy. Solid #0099FF is the workhorse.

  • Text links and interactive text
  • Secondary buttons and outline buttons
  • Active/selected states (tabs, sidebar items)
  • Form focus rings and input borders
  • Badges and tags (non-status)
  • Toggles, checkboxes, radio buttons when active
The Rule of One

In any given view, there should be at most one gradient element visible at a time (excluding the logo). This is the "hero" element — typically the primary CTA or a section header. Everything else uses solid brand colour or neutrals. If you're debating whether something should be gradient or solid, it should be solid. The gradient is earned, not defaulted to.

Usage Rules

Hierarchy Through Weight

Use weight to create subtle hierarchy within the same size. SemiBold (600) for labels, Regular (400) for values. Bold (700) only for headings.

Negative Letter-Spacing for Large Text

Text above 24px uses -0.02em. Display text uses -0.03em. Body text uses default. Keeps large headings tight and balanced.

Monospace for Data

Use SF Mono / Fira Code for hex values, timestamps, IDs, and code. Never as a primary typeface — only for technical metadata.

No Truncation Without Tooltip

If text must be truncated, a tooltip showing full text is mandatory. Task names should expand their containers before resorting to truncation.

Key Icon Decisions

Agent = Hexagon outline. Mirrors the hexagonal avatar shape for AI agents. Immediately communicates "this is an autonomous entity." The old layers/stack icon was too easily confused with "project" or "database."

Workflow = Branching flow. Two parallel paths merging — represents the dependency graph structure that defines DeepFlow workflows. Distinct from "project" which is a container.

Project = Folder. A project is a container for workflows and tasks. The folder metaphor is universally understood and doesn't conflict with any other icon in the set.

09

Elevation & Depth

Three tiers of elevation. In dark mode, higher = lighter background + more shadow.

Flat
Cards, list items, containers

neutral-900 + 1px border

Raised
Hover states, dropdowns, tooltips

neutral-850 + shadow-md

Floating
Modals, command palette, overlays

neutral-800 + shadow-lg

Elevation Rules

In dark interfaces, shadow alone doesn't convey elevation — the eye reads "lighter = closer." Combine background lightness with shadow intensity. Never use more than 3 levels simultaneously. The page background (neutral-950) is the ground plane.

Motion & Animation

Motion is functional, not decorative. It communicates state changes and provides spatial continuity. Hover over the cards to preview.

Fast — 150ms
cubic-bezier(0.4, 0, 0.2, 1)
Hover states, toggles, small reveals
Normal — 250ms
cubic-bezier(0.4, 0, 0.2, 1)
Panel slides, expand/collapse, status changes
Slow — 400ms
cubic-bezier(0.4, 0, 0.2, 1)
Page transitions, modal enter/exit, graph layout
Motion Principles

Purposeful: Every animation answers "what changed?"
Quick: Default to fast. Slow only for major transitions.
Respectful: Honour prefers-reduced-motion.
Consistent: One easing curve. No bouncing or spring physics.

Graph-Specific Motion

Edge flow: Animated dots along dependency edges. Speed proportional to zoom level.
Node entrance: Fade + scale (0.95→1.0), staggered 30ms per node.
Status change: Brief pulse glow on node (150ms).

Node Structure

Rectangular card with rounded corners and a 3px left accent border coloured by status — part of the Single Line System. Selection State

Selected nodes get a brand-coloured border and a subtle glow ring (3px spread, brand at 15% opacity). This makes the active node immediately visible even in dense graphs.

Zoom Behaviour

At high zoom, show full node detail. At medium zoom, show title + avatar only. At low zoom, collapse to a coloured dot (status colour) with a miniature avatar. Never show empty rectangles.

16

Graph Edges

Dependency lines between nodes. Edges show direction, status of the upstream node, and animated flow to indicate active data movement.

Complete → Ready
Upstream done. Downstream can proceed. Solid line, success colour.
Active Flow
Upstream in progress. Animated dots show live data flow direction.
Pending
Upstream not started. Neutral/muted line. No animation.
Blocked
Upstream failed. Error colour. Downstream tasks visually blocked.
Edge Rules

Edges inherit the status colour of their upstream (source) node. This tells the user "what's feeding into this task?" at a glance. Flow animation (dots moving along the edge) only appears on active edges — never on pending or complete. Edge curves use smooth bezier paths, never sharp angles. All edges flow left-to-right to reinforce sequential reading direction.

Symbol Set Rationale

○ Pending: Hollow circle — empty, nothing there yet.
▶ Ready: Play button — "press play to start." Universal media metaphor.
❚❚ In Progress: Pause bars — actively running, like a media player mid-track.
❗ Needs Approval: Exclamation mark — human intervention, sign-off, or approval gate required. This is a blocker that only a person can resolve.
✓ Complete: Checkmark — done. The most universal symbol of completion.
✕ Failed: Cross — error. Universally understood as "something went wrong."

When to Use What

Status symbols (20-32px): Primary format. Task cards, graph nodes, list rows, detail panels. Always prefer symbols over dots when space allows.
Status dots (8px): Ultra-compact only — minimaps, inline mentions, breadcrumb badges.
Progress bars: Workflow/phase completion percentage. Brand gradient for normal, status colours for complete/blocked.
Stacked status bars/donuts: Dashboard-level project overview for James and Ravi's command centre.

As Action Buttons

Status symbols double as quick-action buttons. A "Ready" task shows ▶ — tap it to start. An "In Progress" task shows ❚❚ — tap it to mark complete (the tick briefly previews on hover). This is the primary way Sarah interacts with tasks: see the status, tap to advance it. One gesture, zero menus.

Why Not Just Dots?

Dots rely solely on colour for meaning. Symbols encode status in colour + shape simultaneously. This means: colourblind accessibility out of the box, distinction from property pills (which use coloured dots), and a natural transition from passive indicator to interactive button.

Auto-Save Indicator Pattern

Auto-save reduces anxiety. Show confirmation with a tick and relative time ("just now", "2 min ago"). On hover, show the full timestamp. Never show "Unsaved changes" without an auto-save — if the system auto-saves, just confirm it did. Inspired by Google Docs, Notion, and Linear which all use subtle save indicators rather than explicit save buttons.

Focus Ring

All inputs use a 3px box-shadow ring in --brand-solid at 12% opacity on focus. This is visible in both light and dark mode without being distracting. Never rely solely on border-colour for focus indication.

Error Pattern

Error state replaces the focus ring with --status-error border and background glow. Always pair with a text message below the field — colour alone isn't accessible. Error messages use the ⚠ icon prefix for scanability.

Permissions & Access Control

DeepFlow serves teams handling regulated, sensitive data. The permission model is hierarchical, level-scoped, and applies equally to humans and AI agents. This section defines how permissions are surfaced in the UI.

The Permission Model

Permissions are granted at a specific level of the project hierarchy and automatically cascade to all sub-levels. A user with access at L2 can see L2, L3, L4 — but not necessarily L1 (the parent). This is critical for regulated environments where a contractor may work on a specific workstream without seeing the full project scope.

L1 ICAAP 2026 ─── Sarah (Owner), James (Read-only)
 ├─ L2 Capital Modelling ─── David (Member), DF Agent (Agent)
 │  ├─ L3 Run Model ─── DF Agent (inherits from L2)
 │  └─ L3 Validate Output ─── David (inherits from L2)
 ├─ L2 Risk Assessment ─── Maya (Member) ← no L1 access
 └─ L2 Board Approval ─── James (Read-only) + approval gate assignee
Permission Roles
O
Owner

Full control. Manage permissions, delete workflow, assign tasks, modify structure. Receives all escalation notifications. Can designate tasks as approval gates and assign specific approvers.

M
Member

Create, edit, and advance tasks. Assign sub-tasks. Cannot modify permissions or delete the workflow. May be assigned as approver on specific gates by the Owner.

Read-only

View tasks, status, and audit trails. Cannot edit, advance, or comment. For executives and external stakeholders who need visibility without interaction.

A
Agent

AI agents inherit the permissions of their logged-in user. Chat and agent actions are scoped to exactly what that user can see and do. No privilege escalation.

Approval Gates (Not a Role)

Approval is not a global role — it's a per-gate permission. An Owner can designate specific tasks as approval gates and assign specific Members, Read-only users, or even external users as approvers for those gates. This is more flexible than a fixed "Approver" role: a compliance officer might approve regulatory sign-offs but not budget decisions, while a finance lead approves budgets but not code reviews. Approval permissions are visible on the task detail panel and in the permission summary.

Permission Boundaries in the UI

Hidden, not greyed out. If a user doesn't have access to a level, they don't see it at all — the node doesn't render on the graph, the row doesn't appear in the table. There's no "you don't have permission" ghost state. This prevents information leakage even at the metadata level.

Read-only mode is visually distinct: action buttons (▶ ✓ ❗) render without hover/click affordance. A subtle Read-only badge appears in the top bar. Edit icons are hidden. The user sees the full picture but can't touch it.

Sharing & Invite Flow

Permission is granted per-level via a "Share" action on any workflow node. The dialog shows:

1. User/team search
2. Role dropdown (Owner / Member / Read-only)
3. Scope indicator: "Access to Capital Modelling and all sub-levels"
4. Current members list with role badges

The scope indicator is crucial — it makes the cascade visible. No hidden side effects.

Notification Escalation — "Find the Human"

When an AI agent fails or a task needs intervention, the notification goes to the human owner of that layer/workflow. But if the owner at that level is also an AI agent, the system escalates upward through the hierarchy until it finds a human owner. Notifications never dead-end at an agent.

✕ L3 task "Run Model" fails
→ L3 owner: D DeepFlow Agent (AI — escalate ↑)
→ L2 owner: D David Okafor (Human — notify ✓)
David receives: "AI execution failed on 'Run Model' — review output and retry or reassign"
AI Agent Permission Scope

The chat/agent sees only what its user sees. When Sarah uses chat to ask "show me blocked tasks," the AI queries only tasks within Sarah's permission scope. When Ravi asks the same question, he gets his own scoped view. There is no "admin AI" — every agent action is bounded by the logged-in user's role and level access.

Audit Trail & Compliance

Every permission change is logged in the audit trail: who granted access, to whom, at what level, with what role, and when. Permission changes are never silent.

For David (compliance officer), the audit trail surfaces a dedicated "Permissions" filter showing all access changes chronologically. This is essential for regulated industries where "who had access to what, when" is an audit requirement.

Dialogs & Panels

Modal dialogs and slide-over panels for confirmations, editing, and approval gates. Button placement follows a strict right-aligned convention with the primary action always rightmost.

Button Placement Rules
Placement Convention
  • ✓ CTA / confirm / success button → RIGHT
  • ✓ Cancel / dismiss / tertiary → LEFT of primary
  • ✓ All buttons are BOTTOM-RIGHT aligned
  • ✓ Dismiss via Cancel button or Escape key
  • ✗ No "X" close button on confirmation/destructive dialogs
Why No "X" on Confirmations?

The X button in the top corner is ambiguous on destructive dialogs — does X mean "cancel the action" or "confirm the deletion"? Removing it forces an explicit choice. Escape key provides the same affordance without the ambiguity.

Rationale — Button Placement & the X Button

The X button in the top corner is a UX antipattern for destructive dialogs — it's ambiguous (does X mean "cancel the action" or "confirm the deletion"?). Bottom-right button placement follows the natural reading flow (left-to-right, top-to-bottom) and matches macOS/iOS conventions. The primary action is always rightmost, making it the last thing the eye hits. This aligns with Apple HIG and Material Design 3.

Honest critique: Removing the X button can cause friction. Many users — especially those from Windows backgrounds — instinctively reach for the top-right corner to dismiss. Removing it entirely may feel "trapped." This is a real usability cost.

Compromise: The X button is allowed on non-destructive panels (task detail view, settings, info overlays) where dismissal has no consequences. It is prohibited on confirmation dialogs (delete, approve, reject) and any dialog with a form that could lose data. This gives users the familiar escape hatch where it's safe, while forcing deliberate choice where it matters.
