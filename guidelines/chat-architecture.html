Architecture Specification

Non-Blocking Chat Architecture

DeepFlow's chat is not a chatbot. It's a command layer — fast for status, smart for planning, and never, ever blocked. Three layers. One seamless experience.

⚡ TL;DR — 30 Second Summary
  • Chat uses a three-layer architecture: fast responder (<2s), heavy planner (~5s), and persistent world model (graph DB)
  • Never blocks. You can ask "what's blocked?" while a report is being generated. The fast layer answers instantly; the heavy layer works in the background
  • Non-trivial requests become tasks in the project graph — not inline chat responses. "Book a meeting" creates 6 linked tasks, not one reply
  • Tasks execute in complete isolation — only see their description + upstream inputs. Task description quality is critical
  • Chat never sends emails, creates calendar events, or calls external APIs directly. It creates tasks. Task executors have the tools. Safer, auditable, reversible
  • Proactive gap-filling: chat doesn't just respond — it scans for stalled tasks, missed deadlines, and context gaps on a configurable heartbeat
  • User configures authority rules: what chat can auto-execute vs. what needs approval
THE PROBLEM

Ravi's Day

Why single-threaded chat is broken for the people DeepFlow serves.

It's 10:47 AM. Ravi is in a cab between meetings. He has 12 minutes before his next call. His phone buzzes — WhatsApp from a client, two unread LinkedIn InMails, an email from his accountant, and a Slack ping from the contractor he hired last week. He's already mentally composing a reply to the WhatsApp while skimming the email subject lines.

Ravi runs a solo consultancy. He is the CEO, the project manager, the analyst, and the client-facer. He juggles five systems — WhatsApp, email, CRM, LinkedIn, MS Word — and context-switches every five minutes. That's not a complaint; it's his operating model.

He opens DeepFlow on his phone. He needs two things: "What's blocked in ICAAP?" and "Set up that review meeting with Sarah." Two requests. Maybe 15 seconds of his attention.

But here's the problem. He asked DeepFlow to "write the ICAAP executive summary" earlier this morning. That task is still generating. The chat is blocked. A spinning indicator. He can't type. He can't ask about status. He can't give new instructions. The entire interface is frozen on a single long-running response.

That's unacceptable. For someone juggling five systems in a 12-minute window, a chat that blocks for 60 seconds isn't just slow — it's useless. Ravi closes the app and sends a WhatsApp instead.

The single-threaded chat model assumes the user will wait. Ravi never waits. He dips in, checks status, fires instructions, and moves on. DeepFlow's chat must work the way Ravi works — instant for the simple stuff, asynchronous for the heavy stuff, and never, ever blocked.

ARCHITECTURE

Three-Layer Chat Architecture

Every message hits Layer 1 first. Layer 1 answers what it can, routes what it can't, and the user never waits.

⚡
Layer 1: Fast Responder
Lightweight model · Always-on · Instant
< 2 SECONDS

Handles: status queries, quick answers, task capture, context inference, trivial acknowledgements, and routing decisions.

Layer 1 is a lightweight model that sits between the user and everything else. It has read access to the project graph, calendar, and task history. It answers what it can directly — no task created, no heavy model invoked. If the request is non-trivial, it routes to Layer 2 immediately and tells the user it's being handled.

Example 1 — Status Query
"When's my next 30min slot?"
→ Reads calendar directly. "You're free 2:00–2:30 PM today."
Example 2 — Project Status
"What's blocked in ICAAP?"
→ Reads project graph from DB. "Stress testing is blocked by capital model (72% complete, running now). Everything else is on track."
Example 3 — Routing to Layer 2
"Book a 45min meeting with Sarah today"
→ Detects non-trivial multi-step request. Routes to Layer 2. Immediately replies: "Setting that up now…"
🧠
Layer 2: Planner
Heavy model · Called on demand · Creates task DAGs
~ 5 SECONDS

Handles: writing detailed task descriptions, decomposing requests into task DAGs, matching workflow templates, deciding execution model per task, checking calendars and availability.

Layer 2 is not always running. It's invoked by Layer 1 when a request needs planning. It has access to the world model (Layer 3) and creates structured, executable task graphs. The user is never waiting for Layer 2 — they got a fast acknowledgement from Layer 1 and can continue chatting.

Example — Meeting Workflow
"Book a 45min meeting with Sarah today"
→ Layer 2 checks Ravi's calendar + Sarah's availability
→ Creates a meeting project from template:
  Task 1: Find available slot → auto-execute
  Task 2: Send calendar invite → auto-execute
  Task 3: Create prep doc from ICAAP status → auto-execute
  Task 4: Set up recording → fires at meeting time
  Task 5: Post-meeting: transcribe + summarise → fires after
  Task 6: Send notes to Sarah → needs approval
  Task 7: Add action items to ICAAP → needs approval
Example — Research Task
"Research competitor pricing in UK market"
→ Layer 2 writes a detailed task description with expected output format
→ Assigns to AI agent
→ Fires execution
→ User gets notification when research is complete
🌐
Layer 3: World Model
Persistent · Stored in graph DB · Always updating
PERSISTENT

Contains: a map of Ravi's projects, people, commitments, preferences, and working patterns. This is not a chat context window — it's a structured database that outlives any conversation.

NOT stored in model context. Stored in DeepFlow's graph database. This means it doesn't degrade as conversations get longer, doesn't get "forgotten," and can be queried, updated, and audited independently.

Updated by: task outputs (when a research task completes, its findings update the world model), chat conversations (when Ravi mentions a new contact or preference), and proactive questions (chat asks "Is Sarah still on the board review?" to keep the model accurate).

Chat reads from the world model to infer context, answer questions, and detect gaps. It's what allows Layer 1 to know that "the meeting" means "the ICAAP review with Sarah on Thursday" without being told.

EXECUTION

Execution Isolation

Each task executes in a sealed environment. It sees only what it needs. This makes the system safer, more auditable, and more parallelisable.

A task executor is like a contractor you hand a brief to. It gets the description, the inputs from upstream tasks, and the hub-level resources. Nothing else.
✅
What a task executor sees

Its own task description (written by Layer 2) — detailed, specific, with expected output format. Inputs from upstream dependencies — files, data, context from completed tasks. Hub-level resources — documents and references attached to the project. That's it.

🚫
What a task executor does NOT see

Chat history. Other tasks' internals. The user's full context. Sibling tasks' working state. Anything outside its explicit scope. This isn't a limitation — it's a feature.

⚠️ Implication: Task Description Quality Is Critical

Because task executors see only their description plus upstream inputs, the quality of that description determines the quality of the output. Layer 2 (the Planner) must write detailed, specific, unambiguous descriptions — because that's all the executor has to work with.

A vague description like "Write the report" will produce vague output. A description like "Write the ICAAP executive summary (2–3 pages, formal tone) covering: capital adequacy assessment, stress test results (from upstream Task 3 output), risk appetite alignment, and board recommendations. Format: Word document, include charts from the capital model output." — that produces something useful.

Output Flow

When a task completes: a summary is returned to the chat panel (user sees "Research is done. 3 competitors identified."). The full output is written to the task's canvas (Workspace). Downstream tasks that depended on this task can now fire — they receive this task's output as their input.

INTELLIGENCE

Context Inference

How chat knows what you're talking about — without you explaining it.

Chat infers the user's intent from three signals, weighted in this order:

👁️
1. What's on screen

Active project, selected node, open canvas tab. If Ravi is looking at the ICAAP graph with the "Risk Assessment" node selected, "what's the status?" means that node.

💬
2. Recent conversation

Last few messages in the chat thread. "Also add slides for that" — "that" refers to the meeting just discussed.

⚙️
3. Running tasks

Currently executing tasks and sub-agents. "Is it done yet?" likely refers to the task that's actively running.

Confidence Threshold: 80%

If chat is 80%+ confident about what the user means, it assumes the context and acts. Below 80%, it asks — but concisely, with options:

💬 "Are you asking about the ICAAP workflow (on screen) or the competitor research (running in background)?"

Not a paragraph of explanation. Just the choices. Respect the user's time.

SECURITY

Chat Tools vs Task Executor Tools

A deliberate separation. Chat reads and creates. Task executors act. This boundary makes the system safer, more auditable, and reversible.

💬 Chat Agent Read + Create
  • Read DB — projects, tasks, status, history
  • Create and update tasks and projects
  • Query calendar and search
  • Ask the user questions
  • Create notifications
💬 Chat Agent Does NOT have
  • Email sending
  • Calendar event creation
  • External API calls
  • File generation
  • Web browsing
⚙️ Task Executors Per task type
  • Email — send meeting notes, share reports
  • Calendar — create/modify events
  • Document generation — Word, PDF, slides
  • Web search/browsing — research tasks
  • Code execution — data analysis
  • Image generation — brand tasks
  • External APIs — CRM, LinkedIn, etc.
Why This Matters

Chat never sends an email directly. It creates a task: "Send meeting notes to Sarah." A task executor with email tools picks it up, executes it, and marks it done. This means every external action is:

Safer — chat can't accidentally email the wrong person. The task description specifies the recipient.
Auditable — every email sent has a task record: who requested it, when, what was sent, approval status.
Reversible — tasks that need approval can be reviewed before execution. You can't un-send an email, but you can reject a task.

PROACTIVE

Proactive Gap-Filling

Chat doesn't just respond. It actively maintains the world model, fills in gaps, and surfaces what needs attention — before the user asks.

⚡ Trigger-Based (Immediate)

Fired by events — task completion, meeting end, upstream dependency resolved.

Meeting ended → "Want me to transcribe and summarise?"
Task output ready → "Competitor research is done. [View report →]"
Upstream complete → "Capital model is done. Ready to start the executive summary?"
💓 Heartbeat-Based (Periodic)

User-configurable interval: 30 / 60 / 120 minutes. Scans for things that need attention.

Stalled task → "Stress test hasn't progressed in 3 days. What's the situation?"
Upcoming deadline → "ICAAP board meeting is Thursday. Prep doc isn't started."
Offline changes → "Sarah was added to the project but has no tasks. Assign the risk review?"
🔧 User-Configurable Rules
⏱ Heartbeat frequency: 30 / 60 / 120 min
🔔 Which prompt types to receive
🌙 Quiet hours (no proactive prompts)
🤖 Authority level: auto-execute certain task types?
GOVERNANCE

User Authority Rules

The user controls what chat can do autonomously and what requires explicit approval. Sensible defaults. Full configurability.

Action Default Configurable
Create tasks Auto ✓
Execute trivial tasks Auto ✓
Send emails/messages Ask first ✓
Modify calendar Ask first ✓
Share externally Ask first ✓
Update task status Auto ✓
Add team members Ask first ✓
Delete tasks Ask first ✗ (always ask)
