Component Specification

The Hub Bar

The collapsible workflow control centre. One glance tells you if the workflow is healthy. One click gives you the full picture — stats, team, resources, metadata.

Philosophy

Glanceable Status

The Hub Bar answers one question instantly: "Is this workflow on track?" Collapsed, it's a 36px strip of vital signs. Expanded, it's a pull-down control panel — never a separate page.

Collapsed = heartbeat monitor. Expanded = operating theatre. The user chooses the depth.
36px of Truth

The collapsed bar shows progress percentage, status counts, and due date. Every data point earns its pixels. Sarah glances at this 50 times a day — it must be instantly readable.

Pull-Down, Not Pop-Up

Expanding the Hub Bar pushes the canvas down — it doesn't float over content. The user maintains spatial context. The graph is still visible below; they haven't left the view.

Single Line System

The bottom accent line (brand gradient) signals interactivity. On hover, the line brightens to full opacity — a subtle invitation to click. Expanded, the gradient line frames the bottom of the panel.

Collapsed State

36px — The Vital Signs

Always visible in Graph View and Workspace. Sits between Top Bar and content area. Every element is chosen because Sarah needs it at a glance.

464px — Two-column layout, resources row below. Donut chart hidden to save space.
📊 Stats Column
Donut chart (total task count in centre) + per-status horizontal bars. The donut gives spatial proportion; the bars give exact numbers. Both are needed.
👥 Team Column
Each team member: online indicator (green dot / grey dot) + avatar + name + role tag. AI agents use hexagonal avatars per the design system. "Add team member" opens a search modal.
📎 Resources Column
Workflow-level files that get summarised into each task. Click a resource → opens in the canvas as a review tab. Upload button triggers a dropzone. Below: workflow metadata (due date with countdown, owner, description preview).
Responsive Layout
At 824px: 3-column grid. At 464px: 2-column (stats + team) with resources as a full-width row below. Donut chart hides at narrow widths — the bars convey the same data.
Animation
Expand: 250ms ease. Hub Bar height transitions from 36px → ~220px. Canvas pushes down smoothly. Collapse: same timing, reverse. Use max-height + overflow for CSS-only toggle, or framer-motion for layout animation.
Keyboard
H toggles Hub Bar (when graph is focused). Tab into the Hub Bar focuses the chevron, then the team add button, then upload button. Escape collapses if expanded.

Design Guidelines

Do & Don't

✓ Do
  • Use the collapsed bar as the default — most users check it dozens of times without expanding
  • Make status counts clickable (filter the graph to that status)
  • Persist expanded/collapsed state per user — some users always want it open
  • Update all counts in real-time via WebSocket — never stale
  • Use the brand gradient accent line — this is the interactivity signal
  • Hide lower-priority elements at narrow widths (progressive truncation)
✗ Don't
  • Don't make the expanded view feel like a dashboard page — it's a quick-reference panel
  • Don't show more than 4–5 team members in the expanded list — link to "View all" if more
  • Don't auto-expand on page load — respect the saved state
  • Don't put edit controls in the collapsed bar — it's for reading, not editing
  • Don't use full borders on the Hub Bar — use the Single Line system (bottom accent only)
  • Don't animate the donut chart on every data change — only on initial expand
Implementation

Component Architecture

Component Structure
components/shell/HubBar/
├── HubBar.tsx              ← Main wrapper — reads layout.hubBarExpanded
├── HubBarCollapsed.tsx     ← 36px strip — always rendered
├── HubBarExpanded.tsx      ← Pull-down panel — mounted when expanded
├── HubBarStats.tsx         ← Donut chart + status bars
├── HubBarTeam.tsx          ← Team list + add member
├── HubBarResources.tsx     ← File list + upload + metadata
├── useHubBarData.ts        ← Hook: derives all stats from tree store
└── hubbar.test.tsx         ← Unit tests
Zustand Selectors (useHubBarData)
// All data derived from the single Zustand tree store.
// No separate API calls. O(n) scan of nodes map on change.
// Memoised via Zustand selectors + useMemo.

interface HubBarData {
  workflowName: string
  totalTasks: number
  statusCounts: Record<TaskStatus, number>
  progressPercent: number
  dueDate: string | null
  isOverdue: boolean
  daysRemaining: number | null
  team: TeamMember[]               // from workflow metadata
  resources: WorkflowResource[]    // from workflow metadata
  description: string
  owner: { name: string; avatar: string }
}
Responsive Breakpoints

≥ 600px content width (3-column): Full expanded layout. Donut + bars | Team list + add | Resources + metadata.

< 600px content width (2+1): Stats + Team side-by-side (no donut), Resources as full-width row below. Metadata inline.

Collapsed: Progressive truncation. Due date, team, resources hide at < 520px.

Accessibility

role="region" with aria-label="Workflow summary".

Chevron: aria-expanded.

Status counts use both colour + number — never colour alone. aria-live="polite" on counts for screen reader updates.
